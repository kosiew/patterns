## ⚙️ Technique A — Flexible Signatures (`*args`, `**kwargs`)

```python
def process(event, *args, **kwargs):
    if kwargs.get("verbose"):
        print(f"Processing {event}")
    return event.upper()
````

## 🧠 Technique B — Explicit Keyword-Only Parameters

```python
def process(event, *, verbose=False, retries=3):
    if verbose:
        print(f"Processing {event}")
    return event.upper()
```

## ⚖️ Tradeoff Summary

### `*args` / `**kwargs`

* Maximum flexibility
* Great for forwarding, wrappers, extensibility
* Weak guarantees → harder to reason about
* Errors show up *later* (runtime surprises)

### Keyword-only parameters

* Explicit, self-documenting API
* Enforced at call site
* Easier to maintain and refactor
* Less flexible for unknown future arguments

## 🧠 When to Use What

Use `*args` / `**kwargs` when:

* Writing wrappers, decorators, adapters
* Forwarding arguments to other functions
* Building extensible frameworks (plugins, hooks)

Use keyword-only params when:

* Designing a public API
* You want clarity and constraints
* Parameters have semantic meaning (not just pass-through)

## 🔥 Subtle but Critical Difference

* `**kwargs` says:

  > "I accept anything (and will figure it out later)"

* Keyword-only says:

  > "These are the only knobs you're allowed to turn"

That’s *open-world* vs *closed-world* design.

## ⚠️ Common Misunderstanding

(`**kwargs` trap)

Using it as a lazy escape hatch:

```python
def save(data, **kwargs):
    # what is allowed here?
    ...
```

This destroys:

* autocomplete
* type checking
* readability

...and spreads ambiguity through your system.

## 💡 Elite Secret

You can combine both safely:

```python
def process(event, *, verbose=False, **kwargs):
    if kwargs:
        raise TypeError(f"Unexpected arguments: {kwargs}")
    ...
```

Why this is powerful:

* You keep forward compatibility
* But *fail loudly* on misuse

Even better with typing:

```python
from typing import TypedDict

class Options(TypedDict, total=False):
    verbose: bool
    retries: int
```

Now you can validate structure *without losing flexibility*.

## 🔁 Breakthrough Reframe

Stop thinking:

> "How many arguments should this function take?"

Start thinking:

> "How much *freedom* should callers have?"

Function signatures are not syntax—they are **contracts**.
